2022 Dcard Backend Intern 面試分享

Posted on 2022-06-23 In Interview , Dcard

距離 onboard 的時間剩下不到兩個禮拜，前陣子也因為期末報告、助教和研究所的事情弄到沒時間寫文章，所以就用這篇文章來當作還債吧(?)

先自我介紹一下，我目前是就讀台灣科技大學電機研究所，從去年年底開始找實習的機會，剛好 Dcard 在年初的時候發佈了徵人的消息，所以我就二話不說開始往這個方向努力了XD

首先還是要先感謝 Dcard 的各位大大，讓整個面試的過程相當順暢也愉快，能有這個機會真的很感謝 😀


面試過程

以下會分成 作業關 ➡️ 第一階段面試 ➡️ 第二階段面試 ➡️ CEO 面試 來一一介紹


作業關

基本上要進入 Dcard 前後端實習生的面試流程第一步就是要先完成指定的作業，2022的作業內容簡單來說就是使用 Go 搭配 資料庫 / Cache 來完成一個縮網址系統。

縮網址系統最終需要完成以下兩個 API，以及 unit tests。

  1. 輸入一個 URL 和過期時間，回傳一個縮網址

  2. 輸入一個縮網址，假如存在且尚未過期，重新導向至原網址，反之，回傳404

後面會更詳細去介紹。


第一階段面試

當你履歷被審核過之後就會進到第一階段的面試，第一階段是由後端的團隊來面談，也就是關於技術的面試，分成前1個半小時和後半小時，後面會更詳細去介紹。


第二階段面試

這一個階段是由 Dcard 的 Talen Operation Team 來面試，面試的人都很親切，過程也是讓人不會感到壓迫，後面會更詳細去介紹。


CEO 面試

這一關可以當成是跟 CEO 聊天，基本上沒有什麼限制（但我也沒問什麼怪問題就是了），後面會更詳細去介紹。


作業關

這是我第一次體驗到面試需要寫作業，還蠻酷的，但同時也很怕自己考慮的不夠周全所以真的寫得蠻久的XD

以下是作業的內容

Design and implement (with unit tests) an URL shortener using Go programming language.
Criteria:

  1. URL shortener has 2 APIs, please follow API example to implement
    1. A RESTful API to upload a URL with its expired date and response with a shorten URL.
    2. An API to serve shorten URLs responded by upload API, and redirect to original URL. If URL is expired, please response with status 404.
  2. Please feel free to use any external libs if needed.
  3. It is also free to use following external storage including:
    1. Relational database (MySQL, PostgreSQL, SQLite)
    2. Cache storage (Redis, Memcached)
  4. Please implement reasonable constrains and error handling of these 2 APIs.
  5. You do not need to consider auth.
  6. Many clients might access shorten URL simultaneously or try to access with non-existent shorten URL, please take performance into account.

這個題目算是一個很經典的題目，同時實做到了 POST 跟 GET 和 資料庫的功能，最簡單的寫法不外乎就是做出 POST 跟 GET 的 API 並與 Redis 做搭配，使用 Redis 的 expire command 做到過期失效的功能，但基本上這樣的系統馬上就被打爆了XD

這邊附上我自己的作業 GitHub ，裡面介紹的蠻詳細的，有興趣的可以看一下(看完可以動動小手給個Star)


以下分享幾個我覺得蠻重要的地方

  • 縮網址生成的方式

    其實縮網址生成的方式可以很簡單，也可以很困難(就跟哈味一樣)，為了讓避免有人惡意亂試一堆不存在於資料庫的縮網址去拖垮整體系統的速度，因此關鍵是需要設計一個 Checksum 讓系統能在最一開始的階段就擋下一些非法的輸入，但 Checksum 又不能太簡單，避免馬上就被拆穿。

  • 資料庫搭配 Cache

    在上面我寫到了最簡單的寫法，這個寫法一旦系統流量大了一點之後基本上就爆炸了，所以需要資料庫負責儲存大量的資料，被存取的資料才需要放在Cache。

  • 限制流量避免瞬間同一個 IP 短時間的存取

    其實這個就是去年的作業，有興趣的可以去了解一下，我把去年作業的內容加進去這次的作業，做到一個簡單限制流量的功能。

  • 刪除過期的網址

    在最一開始的時候我有提供一個簡單的架構，會使用 Redis 的原因不外乎就是因為可以指定資料過期的時間，但現在把資料都存在資料庫後，就沒有辦法讓資料過期後自動就被刪除的機制，因此設計了一個 Cron Job 的功能，在定期的時間刪除過期的資料。

當然這個系統還可以更好，如果有更好的建議或想法可以發 PR 給我 😁


第一階段面試

這一個階段首先前1個半小時是跟兩位後端工程師進行面試，問的內容包含了

  • 網路的架構
  • Linux系統架構
  • 作業系統的相關問題
  • 資料庫
  • Golang
  • Docker & k8s

最後有一個小小的 Coding Interview，題目是請使用 Go 做出一個能夠以 batch 的方式從 Go channel 中提取出資料並傳入下一個 func 的功能。


💡 舉例 : 假如今天有100筆資料在 Go channel 內，以 batch 為10的情況下將 Go channel 的資料放入 func A 中，那實做出來的流程大致上會是每讀取 10 次 Go channel 的資料後就把那 10 筆資料傳入 A 中，直到 Go channel 中沒有任何的資料。


當 Coding Interview 結束後，就會跟後端的 Team Lead 面談，基本上會以作業為主，分享自己作業的思路，也可以問一些問題，這一個階段是一個相對重要的階段，由於當時疫情還蠻嚴重的所以改成線上面試，雖然沒辦法參觀到 Dcard 辦公室，不過也讓我相對的比較不這麼緊張。


第二階段面試

這一個階段是由 Dcard 的 Talent Operation Team 來面試，這場面試原先是線上面試，但後來他們覺得沒有讓面試者來參觀了解公司太可惜了，所以就改成實體面試，疫情嚴重的情況下還能夠參觀辦公室，真的超感謝他們 🙏

面試的內容主要是會透過問問題的方式來看你是不是適合 Dcard 這間公司，整體過程我覺得不會給人太大的壓力，在等待的時候還有 Dcard 的人過來偷偷加油XD


CEO 面試

最後一階段是 CEO 面試，由於當時辦公室是遠距狀態，所以跟 CEO 面試是在第二階段結束後，在公司使用他們的電腦進行面試。

在面試前一定要準備好問題，因為 CEO 回答的速度很快，沒準備好問題基本上就會直接卡死，問的問題也沒有太多的限制，可以當作跟 CEO 聊天就好，在聊天的過程中覺得能在這麼年輕就創造出如此具有規模和影響力的軟體真的很厲害。

在面試當時還有聊到咖啡，因為我之前有在咖啡廳打工過，剛好那時候 CEO 有在研究咖啡，對咖啡也蠻有興趣的，所以就聊得蠻開心的。


總結

以上就是我面試 Dcard 大致上的流程了，其實在面試的時候就可以感受到他們是很認真地去對待每一個來面試的人，不管你是應徵正職還是實習生，所以在準備面試和寫作業的時候就要更加的認真去對待，在寫作業的時候盡可能的去想到各種會發生的結果，想辦法去解決它，並透過測試去驗證自己的程式不會發生這個問題。

最後還是很感謝面試的人，真的是辛苦他們了，之後進去 Dcard 期許自己能夠學習更多的知識和技術，讓自己變得更強!! 💪
